> 본 문서는 팀 프로젝트 **TimeCapsule**에서 제가 담당한 역할과 기여 내용을 정리한 문서입니다.

## 역할 및 기여
- **역할**: 프론트엔드
- **기여 내용**
  - TimeCapsule 레이아웃 설계 및 구현
  - Fetch API를 활용한 서버와의 데이터 생성, 조회, 삭제 기능 구현
  - 사용자 클릭 및 입력에 따른 동적 UI 처리

## 기술 스택
- **프론트엔드**: HTML5, CSS3, JavaScript

## 트러블 슈팅 및 배운 점

### 1. css 단위 활용을 통해 반응형 웹 처리
- **문제**
  - 서로 다른 크기의 폰트에 vw 단위를 적용했더니, 화면 축소할 때 줄어드는 정도가 달라 시각적으로 어색함
- **해결**
  - 고정 단위인 rem과 정적 단위인 vw를 함께 사용
- **결과**
  - 큰 폰트는 화면이 줄어들 때 시각적으로 더 빠르게 작아지고, 작은 폰트는 더 천천히 작아져 폰트 크기 변화가 자연스러움

### 2. 모듈화 
- **문제**
  - JS 파일을 모듈화하면서, import된 다른 JS 파일이 실행될 때 브라우저에 해당 DOM 요소가 존재하지 않아 null 반환
  - 예시: main.js:8 Uncaught TypeError: Cannot read properties of null (reading 'addEventListener')
- **해결**
  - 해당 DOM 요소가 브라우저에 존재할 때만 실행되도록 if 조건문 추가
- **결과**
  - 요소가 없을 경우에 에러가 발생하지 않고, 정상적으로 작동

### 3. 필터링 로직 리펙토링
- **문제**
  - 서버에서 가져온 데이터를 브라우저에서 filter와 includes 메소드로 처리했는데, 이 경우 데이터가 많을 때 성능이 느려질 수 있음
- **해결**
  - 프론트엔드에서 제목, 작성자, 날짜만 Spring에 보내고, Spring에서 필터링 하도록 변경
- **결과**
  - 기존과 똑같이 정상적으로 작동함

### 4. fetch API 데이터 받기 
- **문제** 
  - fetch API로 데이터를 받아올 때 JSON 데이터가 항상 배열 형태로 반환된다고 생각하여 view.js에서 json[0]으로 접근 
  - 데이터 형식과 접근 방식이 맞지 않아 데이터가 정상적으로 출력되지 않음 
- **해결** 
  - Spring의 return 타입에 따라 JSON 구조가 다른 것을 알게 되었고, Letter 객체 하나만 반환하므로 배열이 아닌 JSON 객체에 직접 접근하도록 코드를 수정 
- **결과** 
  - 정상적으로 데이터가 출력됨

### 5. fetch API 응답 처리
- **문제**
  - fetch API는 서버가 오류 코드를 반환해도 catch문으로 이동하지 않고, 다음 코드가 실행되어 정상 응답으로 처리됨
- **해결**
  - 조건문과 response.ok 값을 활용하여 정상 응답인지 확인하도록 로직 수정
  - 정상 응답이 아닐 경우, throw new Error()를 사용해 예외를 발생시켜 catch문으로 이동하게 함
- **결과**
  - 오류 응답을 예외 처리할 수 있게 해 안정적인 데이터 처리가 가능

### 6. date 값 전달
- **문제**
  - Spring에 date 값이 빈 문자열로 전달될 경우, 날짜 타입으로 변환하는 과정에서 에러가 발생
- **해결**
  - date 값 존재 여부를 확인 후, 값이 있을 때만 date 값을 전달하도록 if-else문을 사용
- **결과**
  - 에러가 발생하지 않고, 정상적으로 처리됨

### 7. 페이지 이동 문제
- **문제**
  - input 값의 빈칸을 검증하는 로직에서 a 태그로 인해 검증 전에 다음 페이지로 이동하는 문제가 발생
- **해결**
  - event.preventDefault()를 사용하여 a 태그의 페이지 이동을 방지
  - input 값이 모두 있을 경우, window.location.href를 사용해 다음 페이지로 이동
- **결과**
  - input 값 검증 후에 페이지가 이동하도록 제어

### 8. 비밀번호 입력 형식 제한
- **문제**
  - 비밀번호 입력에 숫자 외의 문자도 입력 가능
- **해결**
  - 정규표현식과 test() 함수를 이용해 입력된 문자열이 숫자로만 이루어졌는지 확인
- **결과**
  - 숫자만 입력 가능하도록 제한

### 9. 페이지 간 데이터 전달 문제
- **문제**
  - import문으로 다른 JS를 실행할 때 현재 페이지의 DOM만 존재하기 때문에, 다른 페이지의 DOM 요소는 브라우저에서 찾을 수 없어 null로 나타나 정상적으로 실행되지 않는 문제가 발생
  - main.js에서 View.view(data.id) 방식으로 id를 전달하여 특정 데이터를 조회하도록 구현
  - load_item() 함수가 view.js에만 존재하여 main.js에서 호출할 수 없어 정상적인 동작이 이루어지지 않았음
- **해결**
  - URL에 id 값을 쿼리 파라미터 형식으로 전달하도록 수정
  - window.location.search로 현재 URL의 쿼리 파라미터를 가져온 뒤 URLSearchParams()를 사용해 id 값을 추출하여 관련된 데이터만 조회하도록 로직 변경
- **결과**
  - URL을 통해 전달된 id 값을 이용해 해당 데이터만 정상적으로 조회

## 느낀 점
- TimeCapsule은 기존 ERP 프로젝트를 기반으로 보완한 형태여서 새로 배울 게 많지 않을 것이라고 생각했지만, 원하는 기능과 더 나은 로직을 구현하기 위해 자료를 찾아보면서 다양한 것을 배울 수 있었음
- 학과 공부만 열심히 한다고 프로젝트를 잘할 수 있는 것은 아니며, 프로젝트에서 원하는 기능 구현을 위해 추가 학습이 필요함을 깨달음
- 프로젝트를 진행하며 동기들이 백엔드를 다루는 모습을 보고 신기하게 느꼈고, 이를 계기로 다른 기술에도 관심을 갖게 되어 현재 Spring 강의를 수강 중
- 3학년 1학기 방학 때는 더 많은 역량과 지식을 익혀 더 다양한 기술을 사용하여 프로젝트를 할 예정
